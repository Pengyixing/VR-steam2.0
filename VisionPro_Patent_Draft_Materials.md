# 发明专利技术交底书：基于空间计算的多源异构数据流中继与沉浸式展示系统及方法

## 一、 发明名称
**基于空间计算的多源异构数据流中继与沉浸式展示系统及方法**
*(可供选择的备选名称：一种跨设备的混合现实全景直播与多空间窗口协同渲染方法 / 基于手部骨骼追踪与局部网络自发现的虚拟现实多维监控系统)*

---

## 二、 核心技术摘要
本发明提出了一种面向空间计算设备（如Apple Vision Pro）的双端协同数据渲染与监控系统。该系统通过引入一个高算力的第一计算节点（Mac中继站）来负责异构数据流（HLS视频流、Web网页流、API数据等）的代理、鉴权与分发。第二计算节点（头显设备）通过局域网自动发现协议与中继站建立连接，并构建一个包含“360°全景底底”与“多路独立悬浮窗口（2D/3D）”并发存在的混合现实空间。
同时，本发明提出了一种基于空间唯一标识符（Source ID）的窗口状态隔离机制，确保多个数据源窗口在空间中独立渲染；并首创性地设计了一种基于手部骨骼追踪矩阵变换的随动式腕部交互菜单，解决了全景沉浸环境下的UI遮挡与操作痛点。

---

## 三、 背景技术与现有痛点
1. **设备算力与网络并发瓶颈**：现有的头显设备在独立处理多个高分辨率数据流（如同时拉取4K全景视频、多路监控视频、复杂Web控制台等）时，往往会遭遇发热、卡顿及网络带宽受限等问题。传统方案要求头显直接与各个数据源进行复杂握手与鉴权，加重了端侧负担。
2. **多任务空间状态污染（核心痛点）**：传统的全景或VR直播应用通常为“独占式”单例模式。当用户在空间中尝试打开多个监控视角或网页时，由于应用层共享全局状态变量，往往会导致“点击新数据源，旧窗口被同步覆盖或刷新”的状态污染问题，无法实现真正意义上的“多维并发监控”。
3. **全景环境下的交互局限**：在全景沉浸环境中，传统的固定位置菜单（如固定在用户正前方的UI）极易被全景画面中的高亮或复杂元素干扰，或者当用户转头时脱离视野，导致操作割裂感。

---

## 四、 详细技术方案（具体实施方式）

本系统分为两大核心模块：**中继服务端（Mac端）** 与 **空间渲染客户端（Vision端）**。以下重点阐述本发明的四个核心创新点：

### 创新点一：基于 Bonjour 协议的局域网自发现与双端中继架构
* **功能描述**：Mac 端作为推流中继站，负责整合外部复杂的 HLS 直播源、摄像头数据及 Web 控制台（如盾构机监控参数等），并处理所有跨域及鉴权逻辑。Vision 端无需手动输入 IP，即可自动发现并连接。
* **技术实现细节**：
  1. Mac 端启动后，利用 Bonjour (ZeroConf) 协议在局域网内广播特定服务类型（如 `_vrsteam._tcp`）。
  2. Vision 端的 `StreamDiscovery` 模块启动监听，捕获到广播包后，解析出 Mac 端的局域网 IP 与端口。
  3. Vision 端通过 HTTP/WebSocket 获取系统清单（Manifest），清单中包含了默认的全景底图流地址及可用数据源（Sources）列表。
  4. **有益效果**：大幅降低了头显端的网络请求复杂度，头显端只需与 Mac 节点保持单一高速通道，降低了功耗。

### 创新点二：空间多窗口并发渲染与状态隔离机制（防止全局状态污染）
* **功能描述**：允许用户在 360° 全景空间中，同时召唤出无数个独立的信息面板（包含 WebView 窗口和 AVPlayer 视频窗口），每个窗口显示不同的数据源内容，互不干扰，且支持独立缩放与空间锚定（Pin）。
* **技术实现细节**：
  1. **摒弃全局单例**：打破传统的“选中的源（`selectedSource`）”驱动全局视图重绘的模式。
  2. **唯一标识符路由**：在底层 UI 框架（如 SwiftUI `WindowGroup`）实例化窗口时，强制传入目标数据源的唯一哈希标识符（`sourceID`，类型为 String）。
  3. **独立数据缓存上下文**：客户端在内存中维护一个 `sourceDetails` 字典字典表。当特定 `sourceID` 的窗口被唤起时，窗口组件（如 `WebViewWindow` 或 `VideoWindow`）仅根据自己的 `sourceID` 去字典中提取对应的 URL、账号、密码及展示参数（Muted、Scale等）。
  4. **有益效果**：实现了真正的空间多任务处理。用户可以在观看“全景施工现场”的同时，左边悬浮“A区域监控视频”，右边悬浮“盾构机实时参数网页”，极大提升了工业监控或复杂直播场景的信息获取效率。

### 创新点三：基于手部骨骼矩阵变换的随动式腕部交互菜单 (WristMenuView)
* **功能描述**：在全景空间中，控制面板（菜单）不固定在物理空间的某个点，而是动态“生长”并吸附在用户的左手小臂/手腕处。抬手即现，放下即隐。
* **技术实现细节**：
  1. 开启系统级手部追踪（`HandTrackingManager`），高频捕获左手腕关节（Wrist Anchor）的实时 4x4 Transform 矩阵数据（包含平移 Translation 与旋转 Rotation）。
  2. **局部偏移算法（Local Offset）**：为防止菜单与手臂模型穿模，通过 `wristTransform.rotation.act(localOffset)` 计算出一个沿手臂法线方向略微浮空的局部偏移量（如 `[0.08, 0.08, 0.0]`）。
  3. **四元数旋转矫正（Quaternion Rotation）**：针对骨骼默认坐标系，通过施加基于 X、Y、Z 轴的特定四元数旋转矩阵（如 `xRot = simd_quatf(angle: -.pi/3, ...)`），使 2D UI 面板能够平贴于小臂平面并微微朝向用户面部。
  4. **可见性联动**：通过监听手腕在视野内的可见性状态，利用平滑动画（Opacity/Scale）实现菜单的渐隐渐现。
  5. **有益效果**：提供了符合人类直觉的“本体感觉”交互，解决了 360° 视频遮挡传统 UI 的问题，用户无需四处寻找菜单。

### 创新点四：嵌套式的空间场景层次结构 (ImmersiveView 渲染架构)
* **功能描述**：构建了一种“内翻法线全景球 + 独立漂浮屏幕 + 随动 UI”的复合三维场景树。
* **技术实现细节**：
  1. 在空间坐标系原点（`0, 0, 0`）生成一个巨大的球体网格（Sphere Mesh），并反转其法线使其朝向内部。
  2. 将 HLS 直播流的 `AVPlayer` 绑定至 `VideoMaterial`，并涂装于球体内壁，作为系统的“沉浸式底图”。
  3. 当用户通过腕部菜单请求开启子屏幕时（`spawnScreen`），系统动态生成具有物理碰撞体积（CollisionComponent）和输入捕获能力（InputTargetComponent）的 `FloatingScreenEntity` 实体，并将其作为子节点（Child）挂载到空间中。
  4. **有益效果**：打破了 2D 窗口与 3D 沉浸空间的隔阂，形成了一套完整的“宏观环境感知 + 微观细节交互”的渲染管线。

## 五、 系统流程说明（供画流程图参考）

1. **系统初始化**：Mac端启动，加载源配置清单 -> 启动HLS本地代理服务 -> 通过Bonjour广播服务。
2. **端侧握手**：头显端启动 -> 监听局域网 -> 发现Mac服务IP -> 拉取Manifest和Sources数据。
3. **进入沉浸空间**：头显端渲染巨型反法线球体 -> 播放全景底图HLS流。
4. **触发交互**：头显端开启手部追踪 -> 用户抬起左手 -> 系统计算手腕Transform矩阵 -> 在计算得出的空间坐标处渲染腕部菜单。
5. **多窗口并发实例化**：用户点击菜单中的源A -> 系统传递源A的唯一标识符（ID_A）并分配内存域 -> 渲染源A的独立悬浮窗；用户继续点击源B -> 系统传递源B的唯一标识符（ID_B）并分配新内存域 -> 渲染源B的独立悬浮窗。二者状态互不干涉。

## 六、 本发明带来的有益效果总结
1. 创新性地提出了一种**基于标识符哈希寻址的空间窗口隔离方法**，彻底解决了混合现实设备中跨应用多源并发渲染时的数据状态污染问题。
2. 将重度计算与鉴权转移至中继节点，实现了一种**高低算力设备协同**的瘦客户端渲染架构，降低了头显发热。
3. 提供了一种**空间随动式腕部交互范式**，利用矩阵运算实现了高度贴合人体的自然交互，显著提升了在 360° 全景环境下的操作便利性。
